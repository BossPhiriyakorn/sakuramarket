# แนะนำ Audit Log — การจองล็อค (ยังไม่ implement)

เอกสารนี้เป็น **คำแนะนำ** สำหรับการเพิ่ม audit log ฝั่งการจองล็อค ไม่ได้ implement ในโค้ดปัจจุบัน

---

## 1. บันทึกอะไรบ้าง

| เหตุการณ์ | ข้อมูลที่ควรบันทึก |
|-----------|---------------------|
| **ลูกค้าจองล็อคสำเร็จ** | user_id, room_id, slots (หรือ parcel_id หลังสร้าง), จำนวนช่อง, ยอดชำระ (เหรียญ หรือ 0), เวลา (created_at) |
| **ลูกค้าอัปเดต hold (เลือกช่อง)** | ไม่จำเป็นต้อง log ทุกครั้ง — ถ้าต้องการแค่สรุปว่า user_id ใช้การ “เลือกช่อง” บ่อยแค่ไหน อาจรวมกับ rate limit หรือ log เฉพาะเมื่อมี error |
| **แอดมินจองที่ให้ร้าน (CMS)** | admin_id, registration_id / shop_id, room_id, slots, เวลา — เพื่อเทียบกับฝั่งลูกค้าจองเอง |

---

## 2. เก็บที่ไหน

- **ตารางใหม่ใน DB** เช่น `parcel_booking_audit`:
  - `id`, `created_at`
  - `actor_type`: `'user'` | `'admin'`
  - `actor_id`: user_id หรือ admin_id (UUID)
  - `room_id`, `parcel_id` (หลังสร้างแล้ว ถ้ามี)
  - `slots` (JSONB) หรือ `slot_count`, `grid_x`, `grid_y`, `width`, `height`
  - `amount_paid`: เหรียญที่หัก (0 ถ้าห้องฟรี)
  - `registration_id` หรือ `shop_id` (ถ้ามี)
  - `outcome`: `'success'` | `'failed'` (ถ้าต้องการบันทึกเคสจองไม่ผ่าน เช่น ช่องไม่ว่าง)

- **การเขียน log:** เรียก insert หลังจองสำเร็จ (และถ้าต้องการ หลังเคสล้มเหลวที่สำคัญ) จาก `bookParcelForUser` หรือจาก API route หลัง `bookParcelForUser` return success

---

## 3. แสดงที่ไหน (CMS หรือตรงไหน)

| จุดที่แสดง | รายละเอียด |
|------------|-------------|
| **เมนู CMS ใหม่: “ประวัติการจองที่” หรือ “Audit จองล็อค”** | หน้ารายการ log ทั้งหมด: filter ตามวันที่, room_id, actor_type (user/admin), user_id — แสดงเป็นตาราง (เวลา, ผู้ทำ, ห้อง, ช่อง/parcel, ยอด, ผลลัพธ์) |
| **ในหน้ารายละเอียดร้าน (CMS ร้านค้า)** | แถวหรือบล็อก “ประวัติการจองที่ของร้านนี้” — แสดงเฉพาะ log ที่เกี่ยวกับ shop_id / registration_id นั้น (ลูกค้าจองเองหรือแอดมินจองให้) |
| **ในหน้าผู้ใช้ (CMS ผู้ใช้)** | แถว “การจองที่โดย user นี้” — filter ตาม user_id เพื่อดูว่าผู้ใช้คนนี้เคยจองอะไรบ้าง |

สรุป: **ให้แสดงใน CMS** เป็นหลัก — หน้า “Audit จองล็อค” สำหรับดูรวม และ/หรือ แสดงในหน้ารายละเอียดร้าน + รายละเอียดผู้ใช้ สำหรับดูตามร้าน/ตามคน

---

## 4. สรุป

- **บันทึก:** การจองล็อคสำเร็จ (และถ้าต้องการ เคสล้มเหลว) ทั้งฝั่งลูกค้าและแอดมิน พร้อม user/admin, room, slots/parcel, ยอด, เวลา
- **เก็บ:** ตาราง `parcel_booking_audit` (หรือชื่อใกล้เคียง) ใน DB
- **แสดง:** ใน **CMS** — หน้า “ประวัติการจองที่” และ/หรือ ในหน้ารายละเอียดร้าน + รายละเอียดผู้ใช้

เมื่อพร้อม implement ค่อยเพิ่มตาราง, การ insert ใน flow จอง, และหน้า/component ใน CMS ตามด้านบนได้
